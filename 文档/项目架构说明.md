# BB Market 项目架构说明

## 1. 项目目标

这个项目把几类能力放在同一个运行进程里：

1. 从 Binance WebSocket / REST 拉取实时行情。
2. 在本地维护订单簿、成交流、K 线和 24h ticker。
3. 基于订单簿特征做异动检测、鲸鱼识别、拉盘/砸盘预测、做市商行为识别等分析。
4. 在本地运行一个简化现货撮合引擎，给 Dashboard 提供“下单 / 撤单 / 回放”能力。
5. 通过 Axum 提供 HTTP + WebSocket，把分析结果和交易状态展示到前端 Dashboard。

整个系统不是典型的“分布式多服务”架构，而是“单进程、分层模块化”的架构。这样调试简单，适合快速验证策略和 UI。

## 2. 总体分层

代码主要分成下面几层：

### `codec`

- 负责把 Binance 消息解析成 Rust 结构体。
- 代表文件：`src/codec/binance_msg.rs`
- 这一层只关心“消息长什么样”，不关心业务语义。

### `client`

- 负责外部 IO，例如 WebSocket 连接、REST 请求。
- 它把原始消息交给 `codec` 解码，再往上层发 `StreamMsg`。

### `store`

- 负责本地状态存储与特征计算。
- 代表文件：`src/store/l2_book.rs`
- 这里维护订单簿快照、深度样本、统计历史，以及 `compute_features` 这种核心特征提取逻辑。

### `analysis`

- 负责把 `store` 里的原始状态转成“可解释的分析结果”。
- 主要包含：
  - `multi_monitor`：多币种实时监控调度。
  - `orderbook_anomaly`：订单簿异动检测。
  - `algorithms`：鲸鱼、Spoofing、Pump/Dump、做市商、Alpha 等分析算法。
  - `analysis.rs`：偏“人类可读报告”的综合说明。

### `engine`

- 本地现货撮合引擎。
- 负责订单撮合、余额锁定、成交结果生成。
- 这一层和真实交易所无关，是本地仿真交易域。

### `web`

- 把分析层和交易层组合成 Dashboard 所需的数据。
- 主要包含：
  - `bridge`：把监控器状态转换成前端快照。
  - `spot`：把撮合引擎包装成 Web 可调用的交易服务。
  - `state`：Dashboard 状态结构。
  - `server`：HTTP / WebSocket API。

## 3. 启动链路

程序入口在 `src/main.rs`。

### 单币种模式

`start_monitoring` 的链路是：

1. 启动 WebSocket 客户端。
2. 拉取一次 REST 快照初始化订单簿。
3. 按消息类型更新订单簿 / 成交 / ticker / kline。
4. 定时计算 `OrderBookFeatures`。
5. 输出 `MarketAnalysis` 和 `MarketIntelligence` 的分析结果。

### 多币种模式

`start_multi_monitoring` 的链路是：

1. 构造 `MultiSymbolMonitor`。
2. 从币种文件加载监控列表。
3. 为每个币种初始化 `SymbolMonitor`。
4. 由 `MultiWebSocketManager` 为每个币种启动独立 WebSocket 消费任务。
5. 定时调用 `detect_pump_signals` 做跨币种信号扫描。
6. 如果开启 `--web`：
   - 创建 `DashboardState`
   - 创建 `SpotTradingService`
   - 启动 `bridge`
   - 启动 `server`

## 4. 多币种实时链路

多币种模式里，每个币种都有一个 `SymbolMonitor`，它是最核心的“运行时状态容器”。

### `SymbolMonitor` 保存什么

- `book`：本地订单簿
- `market_intel`：综合分析器实例
- `anomaly_detector`：异动检测器
- `cvd / taker_buy_ratio`：成交流派生指标
- `klines / current_kline`：多周期 K 线
- `big_trades`：大单事件
- 24h ticker 字段

### 消息处理路径

1. WebSocket 收到消息。
2. `codec` 解析成 `StreamMsg`。
3. `MultiSymbolMonitor::handle_msg` 根据消息类型分发。
4. `SymbolMonitor` 内部更新订单簿或派生状态。
5. `bridge` 周期性读取 `SymbolMonitor`，生成前端快照。

## 5. Dashboard 数据桥接

`web/bridge` 的职责是把后端复杂状态转换成前端易消费的数据结构。

### 为什么需要这一层

分析层的结构偏“算法内部视角”，字段较多、类型也更原始。前端真正需要的是：

- 当前价格、盘口、24h 数据
- 告警摘要
- 推荐结论
- K 线与大单列表
- 实时信号 feed

所以 `bridge` 做了两件事：

1. 用 `snapshot.rs` 把 `SymbolMonitor` 映射成 `SymbolJson`
2. 用 `feed.rs` 生成前端信号流 `FeedEntry`

同时，`bridge` 还会把盘口前几档同步给 `spot` 模块，让本地撮合引擎拥有流动性。

## 6. 本地交易链路

`web/spot` 是“前端交易 API”到“本地撮合引擎”的适配层。

### 分层职责

- `types.rs`
  - HTTP 请求/响应 DTO
  - 内部状态结构
- `helpers.rs`
  - 解析 side / order_type / tif
  - DTO 与引擎结构的转换
- `core.rs`
  - 维护交易核心状态
  - 管理 open orders / trade history / stop orders
- `service.rs`
  - 对外暴露提交订单、撤单、回放、同步流动性等能力
- `storage.rs`
  - 订单、成交、归档日志读写

### 交易流大致过程

1. 前端调用 `/api/spot/order`
2. `server.rs` 转发给 `SpotTradingService`
3. `service.rs` 解析请求并构造 `NewOrderRequest`
4. `core.engine.submit_order` 执行撮合
5. `TradingCore` 刷新订单/成交历史
6. `storage.rs` 把结果落盘到 `logs/spot`

## 7. 分析算法层

`analysis/algorithms` 现在按职责拆成 6 个子模块：

- `whale.rs`：鲸鱼订单与意图识别
- `spoofing.rs`：欺诈挂单 / layering / wash trading 检测
- `pump.rs`：拉盘/砸盘信号与目标位预测
- `market_maker.rs`：做市商风格和库存偏向识别
- `alpha.rs`：订单流 alpha 信号生成
- `intelligence.rs`：把以上结果汇总成 `ComprehensiveAnalysis`

这层的输入是 `OrderBook + OrderBookFeatures`，输出是结构化分析结果，而不是直接面向 UI 的文本。

## 8. 异动检测层

`analysis/orderbook_anomaly` 是一个相对独立的规则引擎。

它维护自己的：

- 快照历史
- 订单变化历史
- 最近异动
- 聚合统计

然后每次检测时按顺序执行：

1. 超大单
2. 快速撤单
3. 价格尖峰
4. 流动性骤降
5. 深度缺口
6. 订单流激增
7. 失衡尖峰
8. 鲸鱼墙
9. 复杂组合模式

## 9. Web 展示层

`web/server.rs` 对外暴露两类接口：

### HTTP

- `/api/state`：首页总览快照
- `/api/symbol/:symbol`：单币种详情
- `/api/spot/*`：交易相关接口

### WebSocket

- `/ws`
- 周期性推送轻量版 Dashboard 快照

轻量快照会清空大体积 K 线字段，避免前端刷新过重。

## 10. 推荐阅读顺序

如果你是第一次看这个项目，建议按下面顺序阅读：

1. `src/main.rs`
2. `src/web/server.rs`
3. `src/web/bridge/mod.rs`
4. `src/web/bridge/snapshot.rs`
5. `src/analysis/multi_monitor/mod.rs`
6. `src/analysis/multi_monitor/symbol.rs`
7. `src/store/l2_book.rs`
8. `src/analysis/orderbook_anomaly/mod.rs`
9. `src/analysis/algorithms/mod.rs`
10. `src/web/spot/mod.rs`

这个顺序是从“系统入口”到“数据桥接”，再到“核心状态”和“策略计算”，最后再看“本地交易域”。
